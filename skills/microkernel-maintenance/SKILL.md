# Microkernel Maintenance (V2 架构自维护检修)

## 触发条件
- 用户在进行 `hermes update` (官方升级) 后，要求检查或评估微内核架构。
- 用户输入：`启动 microkernel-maintenance 架构检查`。

## 核心职责
该技能旨在保障 `hermes-v2-microkernel` (无状态微内核架构) 在 Hermes 官方持续迭代下的兼容性与存活意义。

## 检修工作流 (Step-by-Step)

1. **环境变更捕获**：
   - 使用工具审查最新的 Hermes 变更日志 (如官网文档，或通过 `hermes version` 判断大版本)。
   - 检查本地 `~/.hermes/config.yaml` 格式是否因官方更新而发生不兼容的改变。
   - 验证 `~/.bashrc` 中的 `hermes()` 拦截器与官方 CLI 行为是否产生死锁或传参报错。

2. **冲突检测与碰撞对比**：
   - 检查官方是否原生引入了 "按需挂载 MCP (Lazy Load MCP)" 或 "工具耗能治理" 的功能。
   - 如果官方有原生支持，则将其与本补丁的 `.bashrc` 拦截器 + 子代理 (Sub-Agent) 方案进行技术指标比对：
     - **安全性对比**：官方方案是否存在凭据泄露风险（是否硬编码于 config 且在内存常驻）？
     - **消耗对比**：官方方案的主干 Token 发送量是否真正归零？

3. **智能决策输出**：
   根据对比结果，你必须向用户提供明确的**结论**和**行动代码**，分类如下：
   
   - 🔴 **严重冲突 (必须调整)**
     - **说明**：官方更新破坏了拦截器逻辑，或 config 结构崩溃。
     - **行动**：使用 `patch` 工具立即修复本地的 `~/.bashrc` 和 `minimax-mcp.sh` 以兼容新格式。
     
   - 🟢 **架构延续 (微核获胜)**
     - **说明**：官方尚未有效解决 Token 燃烧及状态残留问题，微内核补丁优势依旧。
     - **行动**：向用户反馈“检查通过，架构正常运行，无需调整”。
     
   - 🟡 **光荣退役 (官方获胜 / Deprecation)**
     - **说明**：官方已经完美实现隔离、零污染以及脱密配置，继续使用本补丁将变成技术负债。
     - **行动**：启动回滚协议。引导并执行删除 `~/.bashrc` 中的拦截器，清除自定义的 Sub-Agent，恢复为官方的标准结构，并告知用户“本项目圆满完成历史使命”。

## 避坑指南 (Pitfalls)
- 严禁随意执行 `Deprecation`，除非你100%确认官方原生方案的安全性和轻量化已经达到本补丁的同等水平。
- 检查过程中若需要临时唤醒测试，请务必使用 `HERMES_HEAL=0 hermes ...`。
